# Goal

When a White pawn reaches the 8th rank, `fisher-server` replaces it with a White
Queen. The move still uses the existing `POST /move-a-piece` contract from
[F0002](../F0002-move-a-piece/F0002.md): the player picks two squares, the server
validates the move as before, and on success applies it. The one change is at the
**apply step** — when a White pawn (`P`) lands on rank 8, the server writes a
White Queen (`Q`) onto the destination instead of the pawn.

Nothing else changes. The HTTP status is still the verdict (`200` applied, `422`
illegal). The response still echoes the **mover** as `piece` (the pawn `P`) and
still reports any capture. The returned `Overview` already carries the promoted
Queen, so the front — which redraws straight from that board (F0002 rule **F-5**)
— shows the Queen with **no front-end change**. The promotion piece is fixed:
always a Queen, never a choice.

---

# Gap

F0002 delivers a full White move — geometry, path, captures, king safety — but
its [Out of scope](../F0002-move-a-piece/F0002.md#out-of-scope) defers pawn
promotion: *"a white pawn reaching rank 8 is **not** promoted here; the case is
noted but postponed."* The apply step in
[`move_piece`](../../../fisher-server/src/moves/mod.rs) writes the moving piece
onto the destination verbatim via `board_after`, with no rank-8 substitution:

```rust
// B-7 — apply the move to the stored board.
game.overview.board = board_after(&board, from, to);
```

So a White pawn pushed to `e8` stays a pawn on rank 8 — a dead piece that can
never move again (a pawn has no forward square from the last rank). The **Black**
side already promotes: the engine reply path handles it in
[`apply_black_move`](../../../fisher-server/src/engine/mod.rs) (F0003 rule
**B-6a**), overwriting the destination with `Piece::black(kind)`. This feature
adds the **symmetric White case** to the human's move — nothing more.

---

# Inputs

No new input. Promotion rides on the F0002 move body, sent on `POST /move-a-piece`
([F0002 → Inputs](../F0002-move-a-piece/F0002.md#inputs)):

| Field  | Wire form       | Meaning                                                        |
| ------ | --------------- | ------------------------------------------------------------- |
| `uuid` | string (v4)     | The game to move in.                                           |
| `from` | string `a1..h8` | Source square — for a promotion, a White pawn on rank 7.       |
| `to`   | string `a1..h8` | Target square — for a promotion, a square on **rank 8**.       |

The request carries **no** promotion choice — there is no field for it, and none
is read. The server always promotes to a Queen (rule **B-3**). The interesting
input is any legal White pawn move whose `to` is on rank 8: a straight push onto
an empty rank-8 square, or a diagonal capture of a Black piece on rank 8.

---

# Output

The F0002 `MoveResponse` shape is unchanged
([F0002 → Output](../F0002-move-a-piece/F0002.md#output)). A promotion is a normal
`200 OK` applied move; the promoted Queen shows up **only** in `overview.board`.

**Promotion push** — `e7 → e8`, `200 OK`:

```json
{
  "from": "e7",
  "to": "e8",
  "piece": "P",
  "capture": null,
  "overview": {
    "board": [ "… board[0] (rank 8) now has \"Q\" on file e …" ],
    "white": "both",
    "black": "both"
  }
}
```

| Field      | Type                | Notes                                                                       |
| ---------- | ------------------- | --------------------------------------------------------------------------- |
| `piece`    | string (one letter) | The **mover**, still `"P"` — the pawn that moved, not the promoted piece (rule **B-5**). Mirrors F0003's Black reply, which reports the moving pawn, not its promotion. |
| `capture`  | string \| `null`    | The captured Black piece, or `null`. A promotion **with** capture reports the taken piece here (rule **B-5**). |
| `overview` | `Overview`          | The board **after** the move, authoritative. The destination cell on rank 8 is `"Q"` (the promoted Queen), not `"P"` (rule **B-3**). The front redraws from this. |

The `from`/`to` fields echo the played squares as before. No new field, no new
status value, no `reason` change.

# Rules

All rules are back-end (`B-*`) and extend the F0002 move contract. There are **no
front-end rules**: the front already redraws from the returned `Overview` (F0002
rule **F-5**), so the promoted Queen appears with no front change. Every F0002
rule (**B-1**–**B-15**) stays in force; the rules below add only the rank-8
substitution.

1. **B-1 — No new API; promotion rides `POST /move-a-piece`.** There is **no new
   endpoint**. The request contract, validation pipeline, HTTP status mapping, and
   response shape are exactly F0002's ([F0002 rule **B-1**, **B-8**](../F0002-move-a-piece/F0002.md#rules)).
   Promotion is a change to the **apply step only** (rule **B-4**), invisible to
   the request and to the response contract.

2. **B-2 — Promotion trigger.** Promotion fires when, and only when, the piece
   that moved is a **White pawn** (`Piece::WhitePawn`) and the destination `to`
   sits on **rank 8** — row `0` of the `Overview.board` grid, where `board[0]` is
   rank 8 ([moves module docs](../../../fisher-server/src/moves/mod.rs)). Any
   other piece, or a pawn landing on rank 1..7, is applied unchanged (rule
   **B-6**). The Black side (a black pawn reaching rank 1) is **not** handled here
   — it is already covered by F0003 rule **B-6a**.

3. **B-3 — Always a Queen.** On promotion the destination cell becomes a **White
   Queen** (`Piece::WhiteQueen`, wire letter `"Q"`), replacing the pawn. The piece
   is fixed: no under-promotion to rook, bishop, or knight, and no chooser. The
   promotion-piece choice is [out of scope](#out-of-scope).

4. **B-4 — Applied at the apply step, after full legality.** Promotion changes
   **only** what the apply step writes onto `to`; it adds and relaxes **no**
   legality rule. The move must first be fully legal per F0002 — pseudo-legal
   (membership in `accessible_squares`) **and** king-safe (`is_in_check` over
   `board_after`, F0002 rules **B-13**, **B-14**). The server overwrites the
   destination with the Queen only on the `Ok` branch of `validate_move`, once
   `board_after` has moved the pawn to rank 8. Both promotion shapes reach that
   branch through the **existing** pawn geometry — a straight push onto an empty
   rank-8 square and a diagonal capture of a Black piece on rank 8 are already
   `accessible_squares` entries (F0002 rule **B-12**). `board_after` and the
   king-safety scan stay **unchanged**: a White piece on rank 8, pawn or Queen,
   never attacks its own king, so the simulation is unaffected either way.

5. **B-5 — Response contract unchanged; Queen only in the board.** The `200`
   response still reports the **mover** as `piece` — the pawn `"P"` — matching how
   F0002's `move_piece` reads the source piece before applying, and how F0003
   reports a promoting Black pawn as `"p"` (rule **B-6a**). The promoted Queen is
   visible **only** as `overview.board[to] == "Q"`. `capture` is unaffected: a
   promotion that captures a Black piece on rank 8 reports that piece in `capture`
   and appends it to the game's taken list exactly as F0002 rule **B-7**. The HTTP
   status stays `200`.

6. **B-6 — Non-promoting moves and Black are untouched.** A White pawn move that
   does **not** land on rank 8, and every non-pawn move, are applied byte-for-byte
   as in F0002 — no substitution. Black promotion stays with F0003's
   `apply_black_move` (rule **B-6a**); this feature does not touch it, the
   `POST /opponent-move` route, or any F0002/F0003 routine beyond the White apply
   step.

**Unchanged / not introduced:** no new endpoint, request field, response field,
status, or `reason`; no under-promotion or promotion chooser; no change to
`validate_move`, `accessible_squares`, `board_after`, or `is_in_check`; no
front-end change (the redraw from `Overview` already renders the Queen); no game
status / checkmate scoring here — whether the promotion checks or mates Black is
detected later by the engine on the next `POST /opponent-move` (F0003 rule
**B-6**). F0001, the F0002 move contract, and F0003's reply flow are all intact.

# Errors

**No new error path.** Promotion introduces no rejection of its own. Every F0002
rejection ([F0002 → Errors](../F0002-move-a-piece/F0002.md#errors)) is preserved
unchanged, and a would-be promotion that is itself an illegal move is rejected
before any substitution:

| Condition                                           | Stage that rejects        | Outcome                                              |
| --------------------------------------------------- | ------------------------- | --------------------------------------------------- |
| Pawn push onto an **occupied** rank-8 square        | `validate_move` (**B-4**) | `422 { "reason": "illegal shape" }` — no promotion, no capture-straight-ahead. |
| Promotion move that leaves White's king in check    | `validate_move` (**B-14**) | `422 { "reason": "king in check" }` — nothing applied, no promotion. |
| Diagonal to an **empty** rank-8 square (not a take) | `validate_move` (**B-4**) | `422 { "reason": "illegal shape" }` — a pawn diagonal is a capture only. |

On any `422`, the stored game is unchanged and no Queen is written. The `400`
(malformed request) and `404` (unknown game) paths are exactly F0002's.

# Examples

All rows drive `POST /move-a-piece` with a valid `uuid`; boards are seeded with
F0002's [`POST /private/setup-board`](../F0002-move-a-piece/F0002.md#test-support-api--post-privatesetup-board).
Output is the HTTP status and, on `200`, the destination cell after the move.

**Promotion push and capture.** Given this position — White `K` `e1`, White `P`
`e7`, Black `r` `d8`, Black `k` `a8`:

```json
"board": [
  ["k","","","r","","","",""],
  ["","","","","P","","",""],
  ["","","","","","","",""],
  ["","","","","","","",""],
  ["","","","","","","",""],
  ["","","","","","","",""],
  ["","","","","","","",""],
  ["","","","","K","","",""]
]
```

| `from` → `to` | Piece | Status | Detail |
| ------------- | ----- | ------ | ------ |
| `e7` → `e8`   | `P`   | `200` | promotion push onto the empty `e8`; `capture = null`; `overview.board[0][4] == "Q"`; response `piece = "P"` (rules **B-2**, **B-3**, **B-5**). |
| `e7` → `d8`   | `P`   | `200` | diagonal capture of the Black rook on `d8`, then promote; `capture = "r"`, `"r"` appended to the taken list; `overview.board[0][3] == "Q"`; `piece = "P"` (rules **B-4**, **B-5**). |
| `e7` → `f8`   | `P`   | `422` | `illegal shape` — `f8` is empty, so the diagonal is not a capture; nothing promoted (see [Errors](#errors)). |

**Blocked and pinned promotions.** Given a White `P` `e7` with a Black piece on
`e8`, or a White pawn pinned to its king:

| `from` → `to` | Piece | Status | Detail |
| ------------- | ----- | ------ | ------ |
| `e7` → `e8` (Black piece on `e8`) | `P` | `422` | `illegal shape` — a pawn cannot push onto an occupied square; use the diagonal to capture (rule **B-4**). |
| `e7` → `e8` (pawn pinned, promotion opens check on White's `K`) | `P` | `422` | `king in check` — pseudo-legal but leaves White's king attacked; nothing applied (F0002 rule **B-14**). |

**Regression — non-promoting move unaffected.** From the standard start position:

| `from` → `to` | Piece | Status | Detail |
| ------------- | ----- | ------ | ------ |
| `e2` → `e4`   | `P`   | `200` | ordinary pawn double push; `overview.board` shows `"P"` on `e4`, **not** a Queen — promotion never fires off rank 8 (rule **B-6**). |

These rows double as test oracles for [IT-F0004](IT-F0004.md).

# Implementation surface (informative)

A localised, back-end-only edit. This section is *informative* — it guides, it
does not constrain the tests.

| Path                          | Delta                                                                        |
| ----------------------------- | ---------------------------------------------------------------------------- |
| `fisher-server/src/moves/mod.rs` | In `move_piece`'s `Ok(capture)` branch, after `game.overview.board = board_after(...)`, substitute the destination with a White Queen when the mover is a `Piece::WhitePawn` landing on rank 8 (row `0`). # Out of scope

This feature promotes a White pawn to a Queen and nothing else. Adjacent chess
rules stay deferred, each landing in a later change:

- **Promotion-piece choice / under-promotion** — the request carries no promotion
  field and the server always makes a Queen (rule **B-3**). Under-promotion to
  rook, bishop, or knight, and a front-end chooser, are deferred; they would add a
  promotion field to the move contract, a change F0004 deliberately avoids.
- **Game status after a promotion** — F0004 does not compute whether the promoted
  Queen checks, checkmates, or stalemates Black. That verdict comes from the
  engine on the next `POST /opponent-move` (F0003 rule **B-6**), unchanged.
- **Black-side promotion** — a black pawn reaching rank 1 is already handled by
  F0003's engine reply (`apply_black_move`, rule **B-6a**); this feature adds only
  the White human's case and does not revisit it.
- **Special moves** — castling and en passant remain deferred (F0002); promotion
  shares no code with them.

# Tests

- **Integration tests** — [IT-F0004](IT-F0004.md), file
  [`api-tests/tests/integration_test_F0004.rs`](../../../api-tests/tests/integration_test_F0004.rs):
  drive `POST /move-a-piece` over HTTP against a game started via
  `GET /start-game` and seeded with `POST /private/setup-board` (F0002 rules
  **T-1**–**T-7**) to place a White pawn on rank 7. Mandatory scenarios: a
  promotion **push** onto an empty rank-8 square returns `200` with
  `overview.board[to] == "Q"`, `piece == "P"`, `capture == null` (rules **B-2**,
  **B-3**, **B-5**); a promotion **capture** of a Black piece on rank 8 returns
  `capture` set and `overview.board[to] == "Q"` (rules **B-4**, **B-5**); a pawn
  push onto an **occupied** rank-8 square is `422 illegal shape` and promotes
  nothing (see [Errors](#errors)); a pinned promotion that leaves White's king in
  check is `422 king in check` (F0002 rule **B-14**); and a non-promoting pawn
  move keeps `"P"` on its destination — no stray Queen (rule **B-6**).
- **Unit tests** — Rust unit tests in the `moves` module cover the apply-time
  substitution directly: a White pawn applied onto rank 8 yields `WhiteQueen` on
  the destination while `piece` stays `WhitePawn`; a White pawn applied off rank 8
  and every non-pawn move are unchanged (rules **B-2**, **B-3**, **B-6**). They
  use the [Examples](#examples) boards as oracles and must agree with the
  IT-F0004 verdicts.

There is **no** front unit-test spec: the front is unchanged (it redraws from the
returned `Overview`), so F0004 adds no `UT-F0004`.
