# Coverage goal

Behavioural end-to-end validation, **through the public API**, that `POST /move-a-piece` on `fisher-server` validates and applies a single **White** move against the position the server holds for a game. The verdict rides the HTTP status; on success the response also carries the authoritative board after the move. Each scenario first seeds a **known position** through the private `POST /private/setup-board` seam (rules **T-1**–**T-7**), so no test hand-plays a chain of moves to reach a capture, a pin, or an in-check board. After setup the game is **"white to move"**, so every move under test is a White move (rule **B-9**). The tests drive the running server over HTTP only — they never call internal functions or inspect server state.

What is asserted (black-box, no internal-state inspection):

- `POST /private/setup-board` round-trips a board and echoes back the now-stored `Overview` — proves **T-4**, **T-6** (the whole-board override took);
- `POST /private/setup-board` rejects a malformed board with `400` and an unknown `uuid` with `404`, minting no game — proves **T-2**, **T-3**, **T-7**;
- a valid pawn push returns `200` with the updated `overview` (source emptied, target filled) and `capture = null` — proves **B-7**, **B-12**;
- a valid diagonal pawn capture returns `200` with `capture` set to the taken black piece and that piece gone from the returned board — proves **B-6**, **B-7**, **B-12**;
- valid knight, king, queen, rook, and bishop moves from a real historical middle-game position (Position #4) each return `200` with the moved piece echoed and the board updated — the knight's `L`-jump ignores the pieces it clears, the queen/rook/bishop rays run over empty squares, and the queen and bishop each report the black piece they take — proves **B-4**, **B-5**, **B-6**, **B-7**, **B-12** (and **B-14**: each is a king-safe move that is applied, not refused);
- an empty or black-occupied source is rejected `422 "not your piece"` — proves **B-2**;
- a target holding a white piece is rejected `422 "own piece on target"` — proves **B-3**;
- a move whose shape fits no piece geometry is rejected `422 "illegal shape"` — proves **B-4**, **B-12**;
- a sliding move through an occupied square is rejected `422 "path blocked"` — proves **B-5**;
- a move that leaves White's king in check — a pinned piece — is rejected `422 "king in check"`, while a king-escaping move stays `200` — proves **B-14**, **B-15**;
- a malformed body (`from == to`, missing field) is `400`, an out-of-range square is `400 "invalid square"`, and an unknown `uuid` is `404 "unknown game"` — proves **B-1**;
- the `POST /move-a-piece` response carries the CORS allow-origin header for the front dev origin — proves **B-10**.

The rules referenced above are defined in [F0002.md](F0002.md#rules); the seed contract is in [F0002.md → Test-support API](F0002.md#test-support-api--post-privatesetup-board).

# System under test

## The two APIs under test

This IT drives two HTTP routes on `fisher-server`. One is the private seed seam that installs a known board; the other is the public move contract, the real subject of the feature. Each test case targets exactly one of them.

| Route | Handler → delegate | What the tests drive | Cases |
|-------|--------------------|----------------------|-------|
| `POST /private/setup-board` | `setup_board_handler` → `game::setup_board` | Private seed seam: install a known board, echo it back (rules **T-1**–**T-7**). | TC-001–003 |
| `POST /move-a-piece` | `move_piece_handler` → `moves::move_piece` (→ `validate_move`, then `is_in_check` over `board_after` at the king-safety gate) | The move contract: validate and apply one White move (rules **B-1**–**B-15**). | TC-004–018 |

`GET /start-game` (F0001) is used only to mint a game to seed against; it stays the sole creator of games (rule **T-7**) and is not itself under test here.

## Harness

The real `fisher-server` Axum application, already running and driven over HTTP. Like IT-F0001, the harness reuses the running process at its base URL (production default `:7200` per [architecture.md](../../global/architecture.md) §6), overridable via `FISHER_SERVER_URL`. The client issues real HTTP requests; a game `uuid` comes from `GET /start-game`.

## Boundary

No mock, no internal-state inspection — the test only observes what the public interface returns (status code, headers, JSON body). Every board assertion walks the returned `overview.board` array; the taken-pieces list stays server-side and is not asserted here (rule **B-7**).

The move request fields:

| Field  | Wire form         | Meaning                                    |
|--------|-------------------|--------------------------------------------|
| `uuid` | string (v4)       | The game to move in — a game F0001 created. |
| `from` | string `a1..h8`   | Source square; must hold a white piece.     |
| `to`   | string `a1..h8`   | Target square; must differ from `from`.     |

## Helpers

- `board_1` / `board_2` / `board_3` / `board_4` — build Position #1, #2, #3, #4 as an 8×8 board in the F0001 `Overview` vocabulary.
- `seed(base, board)` — mints a game with `GET /start-game`, installs `board` via `POST /private/setup-board`, asserts the echo took (rule **T-6**), and returns the game `uuid` ready to move against.
- `square_to_rc(square)` — maps an algebraic square to its `(row, col)` in the F0001 board order (`a1` → row 7, col 0; `board[0]` is rank 8), so a test can read one cell of a returned board.
- `cell(overview, square)` — the piece string on `square` of the overview's board, via `square_to_rc`.

# Test cases

Cases split by API (see [The two APIs under test](#the-two-apis-under-test)): TC-001–003 drive the seed seam `POST /private/setup-board`; TC-004–018 drive the move route `POST /move-a-piece`. Each move case seeds one of the four [Seed positions](#seed-positions) first, then makes its move.

## Seed seam — `POST /private/setup-board`

## TC-IT-F0002-001 — setup-board installs a known position and echoes it back

- **Given:** a freshly minted game from `GET /start-game`, and **Position #2** (capture) to install.

  | Item        | Value                     | Expected                       |
  |-------------|---------------------------|--------------------------------|
  | `e4`        | `"P"`                     | present in the echoed board    |
  | `d5`        | `"p"`                     | present in the echoed board    |
  | `white`     | `"both"`                  | echoed back                    |
  | `black`     | `"both"`                  | echoed back                    |

- **Input:** `POST /private/setup-board` with `{ uuid, board: board_2(), white: "both", black: "both" }`.
- **When:** the client sends the request and deserializes the body into `SetupBoardResponse`.
- **Then:**
  1. status is `200 OK` — proves the seed succeeds for a known game (rule **T-2**); a 5xx would mean the body was malformed;
  2. `overview.board` equals the installed board cell-for-cell — proves **T-4** (whole-board override) and **T-6** (the stored board is echoed);
  3. `overview.white == "both"` and `overview.black == "both"` — proves **T-6** (defaults stored and echoed).

## TC-IT-F0002-002 — setup-board rejects a malformed board

- **Given:** a freshly minted game.

  | Input board                                   | Expected status | Expected body                       | Reason                     |
  |-----------------------------------------------|-----------------|-------------------------------------|----------------------------|
  | a `7×8` grid (a row dropped)                   | `400`           | `{ "error": "invalid board setup" }`| not an 8×8 grid (rule **T-2**) |
  | an `8×8` grid with a cell `"X"`               | `400`           | `{ "error": "invalid board" }`      | cell is not `""` or a valid piece letter (rule **T-3**) |

- **Input:** each row above, `POST /private/setup-board` with the given board and a valid `uuid`.
- **When:** the client sends each request.
- **Then:**
  1. status is `400 Bad Request` — proves the handler rejects a malformed board before touching game state (rules **T-2**, **T-3**);
  2. the body deserializes to the row's stable `error` — proves the two rejection messages are distinct and stable for the caller.

## TC-IT-F0002-003 — setup-board on an unknown game is 404 and mints nothing

- **Given:** a `uuid` that names no game in the registry (a fresh v4 never returned by `GET /start-game`).
- **Input:** `POST /private/setup-board` with that `uuid` and a valid board.
- **When:** the client sends the request.
- **Then:**
  1. status is `404 Not Found` — proves **T-7** (setup-board only overwrites an existing game);
  2. the body deserializes to `{ "error": "unknown game" }` — proves the stable unknown-game message;
  3. a follow-up `POST /move-a-piece` on the same `uuid` is also `404` — proves no game was minted (rule **T-7**); `GET /start-game` stays the sole creator.

## Move route — `POST /move-a-piece`

## TC-IT-F0002-004 — a valid pawn push updates the returned board

- **Given:** a game seeded to **Position #1** (standard opening) via `seed(base, board_1())`.
- **Input:** `POST /move-a-piece` with `{ uuid, from: "d2", to: "d4" }` — the pawn double push from the base rank, both crossed squares empty.
- **When:** the client sends the request and deserializes the body into `MoveResponse`.
- **Then:**
  1. status is `200 OK` — proves the move is applied (rules **B-8**, **B-12**);
  2. `piece == "P"` and `capture == null` — proves the moved piece is echoed and a push takes nothing (rules **B-6**, **B-12**);
  3. in the returned `overview.board`, `cell(ov, "d2") == ""` and `cell(ov, "d4") == "P"` — proves the apply step moved the pawn (rule **B-7**);
  4. `overview.white == "both"` and `overview.black == "both"` — proves castling fields are carried unchanged (rule **B-7**).

## TC-IT-F0002-005 — a valid diagonal capture returns the taken piece and removes it

- **Given:** a game seeded to **Position #2** (capture) via `seed(base, board_2())`.
- **Input:** `POST /move-a-piece` with `{ uuid, from: "e4", to: "d5" }` — a forward diagonal onto an enemy pawn.
- **When:** the client sends the request and deserializes the body.
- **Then:**
  1. status is `200 OK` — proves the diagonal capture is legal (rules **B-12**, **B-6**);
  2. `piece == "P"` and `capture == "p"` — proves the taken black pawn is reported (rules **B-6**, **B-8**);
  3. in the returned board, `cell(ov, "e4") == ""` and `cell(ov, "d5") == "P"` — proves the pawn moved onto the target and overwrote the taken piece, so `"p"` is gone (rules **B-7**, **B-6**).

## TC-IT-F0002-006 — an empty or black source is "not your piece"

- **Given:** a game seeded to **Position #1** (standard opening).

  | `from` → `to` | Source content       | Expected                                   |
  |---------------|----------------------|--------------------------------------------|
  | `e4` → `e5`   | empty in the opening | `422 "not your piece"` — no white piece to move |
  | `d7` → `d5`   | black pawn `"p"`     | `422 "not your piece"` — White only (rule **B-9**) |

- **Input:** each row above, `POST /move-a-piece`.
- **When:** the client sends each request and deserializes the body into `IllegalBody`.
- **Then:**
  1. status is `422 Unprocessable Content` — proves the source check rejects a non-white source as illegal, not malformed (rule **B-2**);
  2. `reason == "not your piece"` — proves the precise `IllegalReason` for both an empty source and a black source (rule **B-2**);
  3. the `422` verdict itself proves nothing was applied — an illegal move leaves the game unchanged (rule **B-7**).

## TC-IT-F0002-007 — a white piece on the target is "own piece on target"

- **Given:** a game seeded to **Position #1** (standard opening).
- **Input:** `POST /move-a-piece` with `{ uuid, from: "g1", to: "e2" }` — the knight's `L` lands on the white pawn `e2`.
- **When:** the client sends the request and deserializes the body.
- **Then:**
  1. status is `422 Unprocessable Content` — proves a self-capture is illegal (rule **B-3**);
  2. `reason == "own piece on target"` — proves the precise `IllegalReason`: you cannot take your own piece (rule **B-3**).

## TC-IT-F0002-008 — a move that fits no piece shape is "illegal shape"

- **Given:** a game seeded to **Position #1** (standard opening).

  | `from` → `to` | Piece | Expected                                              |
  |---------------|-------|-------------------------------------------------------|
  | `d2` → `d5`   | `P`   | `422 "illegal shape"` — a pawn cannot advance three squares (rule **B-4**) |
  | `e2` → `d3`   | `P`   | `422 "illegal shape"` — a pawn diagonal onto an empty square is not a capture (rule **B-12**) |

- **Input:** each row above, `POST /move-a-piece`.
- **When:** the client sends each request and deserializes the body.
- **Then:**
  1. status is `422 Unprocessable Content` — proves a shape that fits no piece geometry is illegal (rule **B-4**);
  2. `reason == "illegal shape"` — proves the precise `IllegalReason` for both an over-long push and a non-capturing diagonal (rules **B-4**, **B-12**).

## TC-IT-F0002-009 — a blocked sliding move is "path blocked"

- **Given:** a game seeded to **Position #1** (standard opening), where own pawns sit in front of the back rank.

  | `from` → `to` | Piece | Blocker           | Expected                     |
  |---------------|-------|-------------------|------------------------------|
  | `d1` → `d3`   | `Q`   | white pawn `d2`   | `422 "path blocked"` (rule **B-5**) |
  | `a1` → `a4`   | `R`   | white pawn `a2`   | `422 "path blocked"` (rule **B-5**) |

- **Input:** each row above, `POST /move-a-piece`.
- **When:** the client sends each request and deserializes the body.
- **Then:**
  1. status is `422 Unprocessable Content` — proves a slider stopped by a piece in its path is illegal (rule **B-5**);
  2. `reason == "path blocked"` — proves the precise `IllegalReason`, distinct from `illegal shape`: the shape fits but the path is obstructed (rules **B-4**, **B-5**).

## TC-IT-F0002-010 — king safety rejects a pinned move and allows the king to escape

- **Given:** a game seeded to **Position #3** (pin), where the bishop is pinned to its king along the `e`-file.

  | `from` → `to` | Piece | Expected                                                        |
  |---------------|-------|-----------------------------------------------------------------|
  | `e2` → `d3`   | `B`   | `422 "king in check"` — leaving `e2` opens the file, so `r` on `e8` attacks `K` on `e1` on the post-move board (rules **B-14**, **B-15**) |
  | `e1` → `f1`   | `K`   | `200` — the king steps off the `e`-file to a square no black piece attacks; `capture = null` |

- **Input:** each row above, `POST /move-a-piece` against a game seeded to **Position #3** (re-seed before the second row so it starts from the same position).
- **When:** the client sends each request and deserializes the body.
- **Then:**
  1. the pinned bishop move returns `422` with `reason == "king in check"` — proves the king-safety gate rejects a pseudo-legal move that leaves White's king attacked (rules **B-14**, **B-15**), and that nothing is applied;
  2. the king-escape move returns `200` with `piece == "K"`, `capture == null`, `cell(ov, "e1") == ""` and `cell(ov, "f1") == "K"` — proves a king-safe move is applied (rules **B-14**, **B-7**), so the gate is not a blanket refusal;
  3. together the two rows prove the gate is the final legality step: pseudo-legal but unsafe is illegal, pseudo-legal and safe is legal (rule **B-14**).

## TC-IT-F0002-011 — a malformed body or out-of-range square is 400

- **Given:** a game seeded to **Position #1** (standard opening), so the rejection is the request shape, not the position.

  | Move body                      | Expected status | Expected body                          | Reason                          |
  |--------------------------------|-----------------|----------------------------------------|---------------------------------|
  | `from == to` (`d2` → `d2`)     | `400`           | `{ "error": "invalid move request" }`  | source and target coincide (rule **B-1**) |
  | `to` out of range (`d2` → `d9`)| `400`           | `{ "error": "invalid square" }`        | square outside `a1..h8` (rule **B-1**) |
  | missing `to` field             | `400`           | `{ "error": "invalid move request" }`  | body missing a field (rule **B-1**) |

- **Input:** each row above, `POST /move-a-piece`.
- **When:** the client sends each request and deserializes the body into `ErrorBody`.
- **Then:**
  1. status is `400 Bad Request` — proves the handler rejects a malformed request before any legality check (rule **B-1**), distinct from an illegal move (`422`);
  2. the body deserializes to the row's stable `error` — proves `invalid move request` and `invalid square` are separable and stable for the front.

## TC-IT-F0002-013 — the move response allows the front dev origin (CORS)

- **Given:** a game seeded to **Position #1** (standard opening).

## Valid piece-type moves — seeded from Position #4 (Opera Game)

These five cases each seed **Position #4** (the historical middle game) via `seed(base, board_4())`, then play one legal White move of a given piece type. Each proves that a well-formed, king-safe move of that piece is **applied** (`200`), not refused: the moved piece is echoed, the source empties, the target fills, and captures are reported. Together they extend the valid-move coverage beyond the pawn (TC-004/005) to the knight, king, queen, rook, and bishop.

## TC-IT-F0002-014 — a valid knight move is applied (jumps intervening pieces)

- **When:** the client sends the request and deserializes the body into `MoveResponse`.

  3. in the returned `overview.board`, `cell(ov, "c3") == ""` and `cell(ov, "d5") == "N"` — proves the apply step moved the knight over the intervening pieces without a path check (rules **B-7**, **B-12**, **B-4**).

## TC-IT-F0002-015 — a valid king move is applied (and castling fields are carried unchanged)
