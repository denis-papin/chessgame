# Coverage goal

Behavioural end-to-end validation, **through the public API**, that `GET /start-game` on `fisher-server` creates a game and returns its `uuid` and a well-formed `Overview` board model for the requested mode. The tests drive the running server over HTTP only — they never call internal functions or inspect server state. The board model under test is the project's own `Overview` JSON (an 8×8 array of piece strings plus per-colour castling), **not** FEN.

What is asserted (black-box, no internal-state inspection):

- `200 OK` with a parseable `{ uuid, mode, overview }` body and a valid v4 `uuid` — proves **B-1** (a game is created and identified);
- the default and `mode=standard` responses carry the exact standard `Overview` board and `white="both"`/`black="both"` — proves **B-2**;
- two calls return two different `uuid`s — proves **B-1** (one game per call);
- a `mode=random` layout returns *exactly* the requested `pieces` count for **each** colour (the server randomises positions, not the count) — proves **B-3**;
- a random layout holds exactly one `"K"` and one `"k"` — proves **B-4**;
- every `board` cell is `""` or a single valid piece letter, on an 8×8 array — proves **B-5** and **B-8**;
- no white pawn on rank 8 (`board[0]`) and no black pawn on rank 1 (`board[7]`) — proves **B-6**;
- a random `Overview` has `white="none"` and `black="none"` — proves **B-7**;
- an unknown `mode` is rejected with `400` — proves the [Errors](F0001.md#errors) `invalid mode` contract;
- an out-of-range or non-integer piece count is rejected with `400` — proves the [Errors](F0001.md#errors) `invalid piece count` contract;
- the response carries the CORS allow-origin header for the front dev origin — proves **B-9**.

The rules referenced above are defined in [F0001.md](F0001.md#rules).

# System under test

- **Server:** the real `fisher-server` Axum application, booted by the test harness on a loopback address. The public entry point under test is the `GET /start-game` route (`start_game_handler`, which delegates to `game::start_game` — see [F0001.md → Flow & routines](F0001.md#flow--routines)); the tests only observe its HTTP responses. The harness binds an ephemeral port on `127.0.0.1` and waits until the route answers, so tests never collide on a fixed port. (Production default is `:7200` per [architecture.md](../../global/architecture.md) §6; the test uses whatever port the harness reports.)
- **Client:** an HTTP client (`reqwest::Client`) issuing real `GET` requests to `http://127.0.0.1:<port>/start-game`, optionally with `?mode=` and (for `random`) the `&pieces=` query param. The response body is deserialized into local structs:

  ```rust
  #[derive(serde::Deserialize)]
  struct StartGameResponse { uuid: String, mode: String, overview: Overview }
  #[derive(serde::Deserialize)]
  struct Overview { board: Vec<Vec<String>>, white: String, black: String }
  ```

- **Boundary:** No mock, no internal-state inspection — the test only observes what the public interface returns (status code, headers, JSON body). All `Overview` invariants are checked by walking the returned `overview.board` array in the test, not by reading server memory.

Two helpers live in the test module:
- `all_cells(ov: &Overview) -> impl Iterator<Item = (usize, usize, &str)>` — yields `(row, col, piece)` for the 64 cells, so a test can count, filter by case, or check a specific square.
- `count_color(ov: &Overview, white: bool) -> usize` — counts non-empty cells whose letter is uppercase (white) or lowercase (black).

Because random draws vary, every random case repeats the call **N = 50 times** in a loop and asserts the invariant on each draw, so a single lucky/unlucky layout cannot hide a violation.

# Test cases

## TC-IT-F0001-001 — default call returns the standard starting Overview

- **Given:** a freshly booted server, no games created yet.
- **Input:** `GET /start-game` with no query string.
- **When:** the client calls `client.get(format!("{base}/start-game")).send()` and deserializes the body into `StartGameResponse`.
- **Then:**
  1. the response status is `200 OK` — proves the success path of **B-1**; a 5xx would mean the request was malformed;
  2. `uuid` parses as a v4 UUID (`Uuid::parse_str` is `Ok(_)`) — proves **B-1** (the game is identified);
  3. `mode == "standard"` — proves the default resolves to standard ([Inputs](F0001.md#inputs));
  4. `overview.board` equals the standard 8×8 array — `board[0] == ["r","n","b","q","k","b","n","r"]`, `board[1]` all `"p"`, `board[2..=5]` all `""`, `board[6]` all `"P"`, `board[7] == ["R","N","B","Q","K","B","N","R"]` — proves **B-2**;
  5. `overview.white == "both"` and `overview.black == "both"` — proves **B-2** (standard castling availability).

## TC-IT-F0001-002 — explicit mode=standard returns the same starting Overview

- **Given:** a freshly booted server.
- **Input:** `GET /start-game?mode=standard`.
- **When:** the client sends the request and deserializes the body.
- **Then:**
  1. status is `200 OK`;
  2. `mode == "standard"`;
  3. `overview.board` equals the standard array and `white/black == "both"` — proves **B-2** is reachable explicitly, not only by default.

## TC-IT-F0001-003 — two calls create two distinct games

- **Given:** a freshly booted server.
- **Input:** two sequential `GET /start-game` calls.
- **When:** the client sends both and reads `uuid` from each body.
- **Then:**
  1. both return `200 OK`;
  2. the two `uuid` values differ — proves **B-1** (each call creates a new `GAME`, not a shared one).

## TC-IT-F0001-004 — random returns exactly the requested piece count per colour

- **Given:** a freshly booted server. The piece count is a single caller input applied to both colours, not a server choice — the test sends an explicit `pieces` and checks both colours match it exactly.

  | Query                       | Expected white count | Expected black count |
  |-----------------------------|----------------------|----------------------|
  | `?mode=random&pieces=2`     | `2`                  | `2`                  |
  | `?mode=random&pieces=4`     | `4`                  | `4`                  |
  | `?mode=random&pieces=16`    | `16`                 | `16`                 |
  | `?mode=random` (omitted)    | `16` (default)       | `16` (default)       |

- **Input:** each query row above; the parameterised rows are each repeated N = 50 times (positions vary per draw).
- **When:** for each draw the client parses the body and counts uppercase vs lowercase piece cells with `count_color`.
- **Then:**
  1. every draw returns `200 OK` and `mode == "random"`;
  2. on every draw both `count_color(ov, true)` and `count_color(ov, false)` equal the requested `pieces` (defaulting to `16` when omitted) — proves **B-3** (the server places exactly `pieces` per colour; only positions are random);
  3. across the 50 draws of a fixed `pieces`, the occupied-square sets differ — proves the *positions* are the random part, not the count.

## TC-IT-F0001-005 — random layout has exactly one king per colour

- **Given:** a freshly booted server.
- **Input:** `GET /start-game?mode=random`, repeated N = 50 times.
- **When:** for each draw the client counts cells equal to `"K"` and to `"k"` via `all_cells`.
- **Then:**
  1. on every draw there is exactly one `"K"` cell and exactly one `"k"` cell — proves **B-4**.

## TC-IT-F0001-006 — Overview is a valid 8×8 array of single pieces

- **Given:** a freshly booted server.
- **Input:** `GET /start-game?mode=random`, repeated N = 50 times.
- **When:** for each draw the client inspects the shape and every cell of `overview.board`.
- **Then:**
  1. `board.len() == 8` and every row has `len() == 8` — proves the 8×8 shape of **B-8**;
  2. every cell is either `""` or exactly one character in `{p,n,b,r,q,k,P,N,B,R,Q,K}` — proves **B-5** (one valid piece per cell) and the cell-domain half of **B-8**.

## TC-IT-F0001-007 — random pawns stay off the back ranks

- **Given:** a freshly booted server.

  | Row (rank) | Forbidden cell | Expected |
  |------------|----------------|----------|
  | `board[0]` (rank 8) | `"P"` (white pawn) | never present |
  | `board[7]` (rank 1) | `"p"` (black pawn) | never present |

- **Input:** `GET /start-game?mode=random`, repeated N = 50 times.
- **When:** for each draw the client scans `board[0]` and `board[7]`.
- **Then:**
  1. no cell of `board[0]` equals `"P"` and no cell of `board[7]` equals `"p"`, across all draws — proves **B-6**.

## TC-IT-F0001-008 — random Overview carries no castling rights

- **Given:** a freshly booted server.
- **Input:** `GET /start-game?mode=random`, repeated N = 50 times.
- **When:** for each draw the client reads `overview.white` and `overview.black`.
- **Then:**
  1. `overview.white == "none"` and `overview.black == "none"` on every draw — proves **B-7**.

## TC-IT-F0001-009 — unknown mode is rejected

- **Given:** a freshly booted server.

  | Input query        | Expected status | Expected body |
  |--------------------|-----------------|---------------|
  | `?mode=xyz`        | `400`           | `{ "error": "invalid mode" }` |
  | `?mode=STANDARD`   | `400`           | `{ "error": "invalid mode" }` — mode match is case-sensitive |

- **Input:** `GET /start-game?mode=xyz`.
- **When:** the client sends the request.
- **Then:**
  1. status is `400 Bad Request` — proves the [Errors](F0001.md#errors) contract rejects unknown modes before any game is created;
  2. the body deserializes to `{ "error": "invalid mode" }` — proves the rejection message is stable for the front end.

## TC-IT-F0001-010 — response allows the front dev origin (CORS)

- **Given:** a freshly booted server.
- **Input:** `GET /start-game` with header `Origin: http://localhost:5173`.
- **When:** the client sends the request and reads response headers.
- **Then:**
  1. status is `200 OK`;
  2. the `access-control-allow-origin` header permits `http://localhost:5173` (exact origin or `*`) — proves **B-9**, so the browser front end is not blocked cross-origin.

## TC-IT-F0001-011 — out-of-range or non-integer piece count is rejected

- **Given:** a freshly booted server.

  | Input query                  | Expected status | Expected body | Reason |
  |------------------------------|-----------------|---------------|--------|
  | `?mode=random&pieces=1`      | `400`           | `{ "error": "invalid piece count" }` | below the `2` minimum |
  | `?mode=random&pieces=17`     | `400`           | `{ "error": "invalid piece count" }` | above the `16` maximum |
  | `?mode=random&pieces=abc`    | `400`           | `{ "error": "invalid piece count" }` | not an integer |

- **Input:** each query row above.
- **When:** the client sends the request.
- **Then:**
  1. status is `400 Bad Request` — proves the [Errors](F0001.md#errors) `invalid piece count` contract rejects bad counts before any game is created;
  2. the body deserializes to `{ "error": "invalid piece count" }` — proves the rejection message is stable for the front end;
  3. a valid count at each bound (`pieces=2`, `pieces=16`) still returns `200` — proves the bounds are inclusive (cross-check with TC-004).
