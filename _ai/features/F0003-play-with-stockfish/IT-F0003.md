# Coverage goal

Behavioural end-to-end validation, **through the public API**, that `POST /opponent-move` on `fisher-server` computes Black's reply with the real Stockfish engine and returns the board after that reply plus the game status. The server holds the position for the game, serializes it to a Black-to-move FEN, calls the engine over REST, applies the reply, and scores the result. Each scenario first seeds a **known position** through the private `POST /private/setup-board` seam (the F0002 test-support API), so no test hand-plays a game to reach a mate, a stalemate, or a check. The tests drive the running server over HTTP only — they never call internal functions or inspect server state — and, because `opponent-move` really calls the engine, they run against a live Stockfish container on `:4000` ([architecture.md](../../global/architecture.md) §2.3).

What is asserted (black-box, no internal-state inspection):

- an opening game returns a legal Black reply with `status == "move"` and the board updated (the source square emptied, the target filled with the black piece) — proves the server sent a valid Black-to-move FEN, relayed the engine's move, and applied it (rules **B-4**, **B-5**, **B-6**, **B-8**);
- a reply that captures a hanging white piece returns that piece in `capture` and shows it gone from the returned board — proves **B-8**;
- a reply that checks the White king returns `status == "white-in-check"` — and that verdict is computed by `fisher-server` itself (the engine reports no check), proving the server's own `is_in_check` decides it (rule **B-6**);
- a reply that mates the White king returns `status == "white-in-checkmate"`, read from the engine's `mate 1` score on the reply — proves **B-6**;
- a position where **Black** is already checkmated returns `status == "black-in-checkmate"` with a `null` move and the board unchanged, driven by the engine answering `(none)` — proves **B-3**;
- a position where **Black** is stalemated returns `status == "black-in-stalemate"` with a `null` move and the board unchanged — proves **B-3**;
- a malformed body is `400 "invalid opponent-move request"` and an unknown `uuid` is `404 "unknown game"` — proves **B-1**;
- the `POST /opponent-move` response carries the CORS allow-origin header for the front dev origin — proves **B-10**;
- with the engine unreachable, `POST /opponent-move` returns `502 "engine unavailable"` — proves **B-9** (run only where the harness can stop the engine);
- a real White move (`d2` → `d4` via `POST /move-a-piece`) followed by `POST /opponent-move` composes in one game — White's pawn stays on `d4` while Black's reply is applied on top — proves the front-to-back sequence works end to end (rules **B-4**, **B-5**, **B-6**, **B-8**).

The rules referenced above are defined in [F0003.md](F0003.md#rules); the response shape and the four engine game-cases are in [F0003.md → Output](F0003.md#output) and [F0003.md → Stockfish communication](F0003.md#stockfish-communication); the seed contract is the F0002 test-support API in [F0002.md → Test-support API](../F0002-move-a-piece/F0002.md#test-support-api--post-privatesetup-board).

# System under test

## The API under test

This IT drives one public HTTP route on `fisher-server`, `POST /opponent-move`, plus two fixture seams that only set the position up. Every test case targets `opponent-move`; the seams are never themselves under test here.

| Route | Handler → delegate | What the tests drive | Cases |
|-------|--------------------|----------------------|-------|
| `POST /opponent-move` | `opponent_move_handler` → `engine::opponent_move` (→ `to_fen`, `best_move`, `parse_uci`, `apply_black_move`, then `game_status` with the reply's `mate_in`) | The reply contract: fetch Black's move from Stockfish, apply it, and score the result (rules **B-1**–**B-10**). | TC-001–010 |
| `POST /private/setup-board` | `setup_board_handler` → `game::setup_board` | Fixture seam: install a known board before the reply (F0002 rules **T-1**–**T-7**). Not under test here — proven by IT-F0002. | fixture only |
| `POST /move-a-piece` | `move_piece_handler` → `moves::move_piece` | Fixture seam: play a real White move to set up the position before the reply (F0002 rules **B-1**–**B-15**). Not under test here — proven by IT-F0002. | fixture only (TC-010) |
| `GET /start-game` | `start_game_handler` → `game::start_game` | Fixture seam: mint a game to answer in, and the opening position for TC-001/TC-010. | fixture only |

## Harness

The real `fisher-server` Axum application, already running and driven over HTTP, exactly as in [IT-F0002](../F0002-move-a-piece/IT-F0002.md). The harness reuses the running process at its base URL (production default `:7200` per [architecture.md](../../global/architecture.md) §6), overridable via `FISHER_SERVER_URL`; a game `uuid` comes from `GET /start-game`.

Unlike IT-F0001/IT-F0002, `opponent-move` calls the engine, so these tests also require a **live Stockfish REST container reachable by the server** — the `ghcr.io/samuraitruong/stockfish-docker:14.1` image on `:4000` that `fisher-server` is configured to call (its `ENGINE_BASE`, default `http://localhost:4000`). When the engine is up, the engine-driven cases (TC-001–006) pass; when it is down, `opponent-move` returns `502` — which TC-009 exercises on purpose by stopping the container.

## Boundary

No mock, no internal-state inspection — the test only observes what the public interface returns (status code, headers, JSON body). Every board assertion walks the returned `overview.board` array; the taken-pieces list stays server-side and is not asserted here (rule **B-8**). The engine itself is a real dependency, not a stub: the tests pin the **status** the server derives, not a specific engine move, except where the seeded position makes the reply forced (a mate-in-1, a `(none)`, a single winning capture).

The request carries only the game id:

| Field  | Wire form   | Meaning                                   |
|--------|-------------|-------------------------------------------|
| `uuid` | string (v4) | The game to answer in — a game F0001 created. |

The response fields the tests read:

| Field      | Wire form                         | Meaning                                                        |
|------------|-----------------------------------|---------------------------------------------------------------|
| `from`     | string `a1..h8` \| `null`         | Black's source square; `null` when the game was already over. |
| `to`       | string `a1..h8` \| `null`         | Black's target square; `null` likewise.                       |
| `piece`    | string (lowercase) \| `null`      | The black piece that moved (`p n b r q k`); `null` likewise.  |
| `capture`  | string (uppercase) \| `null`      | The captured **white** piece, or `null` when the reply takes nothing. |
| `status`   | `GameStatus` string               | One of `move`, `white-in-check`, `white-in-checkmate`, `black-in-checkmate`, `black-in-stalemate`. |
| `overview` | `Overview`                        | The board after Black's move — or unchanged when the game was already over. |

## Helpers

- `board_1` / `board_2` / `board_3` / `board_4` / `board_5` — build Position #1–#5 as an 8×8 board in the F0001 `Overview` vocabulary.
- `seed(base, board)` — mints a game with `GET /start-game`, installs `board` via `POST /private/setup-board`, asserts the echo took, and returns the game `uuid` ready to answer against (the same seed helper as IT-F0002).
- `opponent_move(base, uuid)` — issues `POST /opponent-move` with `{ uuid }` and deserializes the body into `OpponentMoveResponse`.
- `square_to_rc(square)` — maps an algebraic square to its `(row, col)` in the F0001 board order (`a1` → row 7, col 0; `board[0]` is rank 8).
- `cell(overview, square)` — the piece string on `square` of the overview's board, via `square_to_rc`.

# Test cases

Each engine-driven case seeds one of the five [Seed positions](#seed-positions) (or, for TC-001, mints the opening), then calls `opponent-move` once and asserts the derived status and board. TC-007–009 cover the request and failure contract, and TC-010 plays a real White move first to prove the full front-to-back sequence.

## TC-IT-F0003-001 — an opening game gets a legal Black reply

- **Given:** a freshly minted game from `GET /start-game` — the F0001 opening, White having conceptually just moved, so it is Black's turn.
- **Input:** `POST /opponent-move` with `{ uuid }` of that game.
- **When:** the client sends the request and deserializes the body into `OpponentMoveResponse`.
- **Then:**
  1. status is `200 OK` — proves the server serialized a Black-to-move FEN, called the engine, and got a reply (rules **B-4**, **B-5**); a `502` here would mean the engine was unreachable;
  2. `status == "move"` — proves an ordinary reply from the opening is neither check nor mate (rule **B-6**);
  3. `from` and `to` are both present, and `piece` is a lowercase black letter — proves the engine's move was parsed and echoed (rule **B-5**);
  4. `capture == null` — proves an opening reply takes nothing (rule **B-8**);
  5. in the returned `overview.board`, `cell(ov, from) == ""` and `cell(ov, to) == piece` — proves the reply was applied to the stored board (rule **B-8**). The exact move is the engine's choice and is not asserted.

## TC-IT-F0003-002 — a reply that captures a hanging piece reports and removes it

- **Given:** a game seeded to **Position #1** (capture) via `seed(base, board_1())`, where the White queen on `d5` is hanging to the Black `e6` pawn.
- **Input:** `POST /opponent-move` with `{ uuid }`.
- **When:** the client sends the request and deserializes the body.
- **Then:**
  1. status is `200 OK` and `status == "move"` — proves the capturing reply is applied and does not check the White king (rule **B-6**);
  2. `capture == "Q"` — proves the taken white queen is reported in the response (rule **B-8**);
  3. `piece == "p"`, and in the returned board the queen's square now holds the black pawn while its origin is empty (`cell(ov, "d5") == "p"`, `cell(ov, "e6") == ""`) — proves the reply moved the pawn onto the target and the taken queen is gone (rule **B-8**). The engine's forced best here is `e6→d5`.

## TC-IT-F0003-003 — a reply that checks White is "white-in-check" (server-decided)

- **Given:** a game seeded to **Position #2** (Black checks White) via `seed(base, board_2())`, where the Black knight on `g5` has the fork `Nf3+`.
- **Input:** `POST /opponent-move` with `{ uuid }`.
- **When:** the client sends the request and deserializes the body.
- **Then:**
  1. status is `200 OK` and `status == "white-in-check"` — proves the server flags a reply that leaves the White king attacked (rule **B-6**);
  2. `capture == null` and `piece == "n"` — proves the checking reply here is a quiet knight move, applied and echoed (rules **B-6**, **B-8**);
  3. the `white-in-check` verdict proves the status is computed by `fisher-server` itself with `is_in_check`, because the engine's `/bestmove` carries no check flag — a plain check has no engine signal (rule **B-6**; see [F0003.md → Stockfish communication](F0003.md#stockfish-communication)). The engine's clear best here is `g5→f3`.

## TC-IT-F0003-004 — a reply that mates White is "white-in-checkmate"

- **Given:** a game seeded to **Position #3** (Black mates White) via `seed(base, board_3())`, where `Qd8-h4` is mate in one.
- **Input:** `POST /opponent-move` with `{ uuid }`.
- **When:** the client sends the request and deserializes the body.
- **Then:**
  1. status is `200 OK` and `status == "white-in-checkmate"` — proves the server reads the engine's `mate 1` score on the reply and reports the White mate without generating any White move (rule **B-6**; see [F0003.md → Stockfish communication](F0003.md#stockfish-communication));
  2. `from` and `to` are present and `piece == "q"` — proves the mating reply was still applied to the board (the game-ending move is played, not suppressed) (rule **B-8**). The forced mate here is `d8→h4`.

## TC-IT-F0003-005 — Black already checkmated returns "black-in-checkmate" with no move

- **Given:** a game seeded to **Position #4** (Black is checkmated) via `seed(base, board_4())`; the board installed is captured before the call for comparison.
- **Input:** `POST /opponent-move` with `{ uuid }`.
- **When:** the client sends the request and deserializes the body.
- **Then:**
  1. status is `200 OK` and `status == "black-in-checkmate"` — proves the engine answered `(none)` and the server, seeing the Black king in check, reported the mate (rule **B-3**);
  2. `from`, `to`, `piece`, and `capture` are all `null` — proves no reply was applied when Black has no move (rule **B-3**);
  3. `overview.board` equals the seeded board cell-for-cell — proves the stored board is left unchanged on a game-over result (rule **B-3**).

## TC-IT-F0003-006 — Black stalemated returns "black-in-stalemate" with no move

- **Given:** a game seeded to **Position #5** (Black is stalemated) via `seed(base, board_5())`; the installed board is captured before the call.
- **Input:** `POST /opponent-move` with `{ uuid }`.
- **When:** the client sends the request and deserializes the body.
- **Then:**
  1. status is `200 OK` and `status == "black-in-stalemate"` — proves the engine answered `(none)` and the server, seeing the Black king **not** in check, reported a stalemate rather than a mate (rule **B-3**);
  2. `from`, `to`, `piece`, and `capture` are all `null`, and `overview.board` equals the seeded board — proves nothing was applied and the board is unchanged (rule **B-3**).

## TC-IT-F0003-007 — a malformed body or unknown game is rejected before the engine

- **Given:** a valid game `uuid` for the malformed row, and a `uuid` that names no game (a fresh v4) for the unknown row.

  | Request body                       | Expected status | Expected body                               | Reason                                   |
  |------------------------------------|-----------------|---------------------------------------------|------------------------------------------|
  | body missing `uuid`                | `400`           | `{ "error": "invalid opponent-move request" }` | request is malformed (rule **B-1**)      |
  | `{ uuid }` naming no game          | `404`           | `{ "error": "unknown game" }`               | no such game in the registry (rule **B-1**) |

- **Input:** each row above, `POST /opponent-move`.
- **When:** the client sends each request and deserializes the body into `ErrorBody`.
- **Then:**
  1. the malformed row is `400 Bad Request` and the unknown row is `404 Not Found` — proves the handler validates the request and looks up the game **before** calling the engine (rule **B-1**), so neither reaches Stockfish;
  2. each body deserializes to the row's stable `error` — proves `invalid opponent-move request` and `unknown game` are distinct, stable messages for the front.

## TC-IT-F0003-008 — the reply response allows the front dev origin (CORS)

- **Given:** a freshly minted game from `GET /start-game`.
- **Input:** `POST /opponent-move` with `{ uuid }` and header `Origin: http://localhost:5173`.
- **When:** the client sends the request and reads the response headers.
- **Then:**
  1. status is `200 OK` — proves the reply is produced under the CORS layer;
  2. the `access-control-allow-origin` header permits `http://localhost:5173` (exact origin or `*`) — proves **B-10**, so the browser front is not blocked cross-origin on the new `POST` route.

## TC-IT-F0003-009 — an unreachable engine returns 502 (run only where the engine can be stopped)

- **Given:** a freshly minted game, and the Stockfish container stopped so the server's `GET :4000/bestmove` call fails. This case runs only where the harness can bring the engine down; elsewhere it is ignored (the other cases require the engine **up**), matching [F0003.md → Tests](F0003.md#tests).
- **Input:** `POST /opponent-move` with `{ uuid }`.
- **When:** the client sends the request and deserializes the body into `ErrorBody`.
- **Then:**
  1. status is `502 Bad Gateway` — proves an engine transport failure surfaces as an engine fault, not a `200` or a `500` (rules **B-5**, **B-9**);
  2. the body deserializes to `{ "error": "engine unavailable" }` — proves the stable engine-failure message, distinct from the `400`/`404` request errors (rule **B-9**).

## TC-IT-F0003-010 — a real White move then an opponent reply compose in one game

- **Given:** a freshly minted game from `GET /start-game` — the F0001 opening, no seed. This is the natural front-to-back sequence: the human plays a White move through the F0002 route, then the front asks for Black's reply.
- **Input:** two calls on the same `uuid`, in order — first `POST /move-a-piece` with `{ uuid, from: "d2", to: "d4" }` (the White pawn double push), then `POST /opponent-move` with `{ uuid }`. The White move's legality is the F0002 contract, already verified by [IT-F0002](../F0002-move-a-piece/IT-F0002.md) (TC-004), kept here only to set up the position.
- **When:** the client plays `d2` → `d4`, then requests Black's reply against the position that move produced.
- **Then:**
  1. the `POST /move-a-piece` call returns `200 OK`, and in its `overview.board` `cell(ov, "d2") == ""` and `cell(ov, "d4") == "P"` — proves the White move applied and was stored on the game (F0002 **B-7**, referenced not re-proven);
  2. the `POST /opponent-move` call returns `200 OK` with `status == "move"` — proves the server serialized the **post-`d4`** position it holds, called the engine, and got an ordinary reply that neither checks nor mates (rules **B-4**, **B-5**, **B-6**);
  3. `from` and `to` are present, `piece` is a lowercase black letter, and `capture == null` — proves a legal, non-capturing Black reply was parsed and applied (rules **B-5**, **B-8**). The exact move is the engine's choice and is not asserted;
  4. in the `opponent-move` response's `overview.board`, `cell(ov, "d4") == "P"` still holds **and** `cell(ov, from) == ""` with `cell(ov, to) == piece` — proves the reply was applied **on top of** the White move, so the two moves compose in the same game rather than resetting it (rule **B-8**).
